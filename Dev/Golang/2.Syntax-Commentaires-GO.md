# Go Syntax et Commentaires

Ce chapitre approfondit la **syntaxe du langage Go** et l'utilisation des **commentaires**, en continuité directe avec les bases déjà vues. Il explique en détail chaque principe, avec des exemples clairs.

---

## 1. Structure générale d’un fichier Go

Un fichier Go est composé des éléments suivants, **dans un ordre précis** :

1. Déclaration du package
2. Importation des packages
3. Fonctions
4. Instructions (statements) et expressions

### Exemple complet

```go
package main

import (
    "fmt"
)

func main() {
    fmt.Println("Hello World!")
}
```

---

## 2. Déclaration du package

```go
package main
```

### Explication

- Chaque programme Go appartient à un **package**
- Le mot-clé `package` est obligatoire
- `main` désigne un programme **exécutable**

**Règle importante** : tout programme exécutable Go doit appartenir au package `main` et contenir une fonction `main()`.

---

## 3. Importation des packages

```go
import ("fmt")
```

### À quoi sert `import` ?

- Permet d'utiliser du code provenant d'autres packages
- `fmt` est un package standard utilisé pour l'entrée/sortie (affichage, formatage)

Exemple :

```go
fmt.Println("Hello World!")
```

Sans l'import de `fmt`, ce code provoquerait une erreur de compilation.

---

## 4. Espaces et lisibilité du code

Go **ignore les espaces et les lignes vides** lors de la compilation.

### Pourquoi utiliser des lignes vides ?

- Améliorer la lisibilité
- Séparer logiquement les parties du code

Même si elles sont ignorées par le compilateur, elles sont fortement recommandées.

---

## 5. Fonctions en Go

```go
func main() {
    fmt.Println("Hello World!")
}
```

### Décomposition

- `func` : mot-clé pour déclarer une fonction
- `main` : nom de la fonction
- `{}` : bloc de code exécuté

Toute instruction placée **à l'intérieur des accolades** sera exécutée.

---

## 6. Instructions (Statements)

```go
fmt.Println("Hello World!")
```

Cette ligne est une **instruction**.

### Séparation des instructions

En Go, les instructions sont séparées par :

- un retour à la ligne (Entrée)
- ou un point-virgule `;`

Exemple équivalent :

```go
fmt.Println("Hello World!"); fmt.Println("Go")
```

Le point-virgule est **ajouté automatiquement** par le compilateur lorsqu'une ligne se termine correctement.

---

## 7. Règle des accolades `{}`

En Go, l'accolade ouvrante `{` **ne peut pas être placée sur une nouvelle ligne**.

### Code invalide 

```go
func main()
{
    fmt.Println("Hello World!")
}
```

Ce code provoque une erreur de compilation.

### Code valide 

```go
func main() {
    fmt.Println("Hello World!")
}
```

Cette règle est stricte et fait partie de la philosophie de Go.

---

## 8. Code compact (non recommandé)

Il est possible d’écrire du code Go très compact :

```go
package main; import ("fmt"); func main() { fmt.Println("Hello World!") }
```

### Pourquoi éviter ?

- Difficulté de lecture
- Mauvaise maintenabilité
- Contraire aux bonnes pratiques Go

Go privilégie un **code clair et lisible**, même s'il est plus long.

# Go Comments

Les commentaires sont des textes **ignorés par le compilateur**.

Ils servent à :
- expliquer le code
- améliorer la lisibilité
- désactiver temporairement du code

---

## 9. Commentaires sur une ligne

Les commentaires sur une ligne commencent par `//`.

```go
// Ceci est un commentaire
fmt.Println("Hello World!")
```

Tout le texte après `//` est ignoré.

### Commentaire en fin de ligne

```go
fmt.Println("Hello World!") // Ceci est un commentaire
```

---

## 10. Commentaires multi-lignes

Les commentaires multi-lignes commencent par `/*` et se terminent par `*/`.

```go
/*
Le code ci-dessous affiche
Hello World à l'écran
*/
fmt.Println("Hello World!")
```

Ils sont utiles pour des explications plus longues.

---

## 11. Commentaires pour désactiver du code

Les commentaires sont souvent utilisés pour **empêcher l'exécution** d'une ligne de code.

```go
func main() {
    fmt.Println("Hello World!")
    // fmt.Println("Cette ligne ne sera pas exécutée")
}
```

Très pratique pour tester ou déboguer sans supprimer du code.

---

## 12. Bonnes pratiques sur les commentaires

- Préférer `//` pour la majorité des cas
- Utiliser `/* */` pour des blocs explicatifs
- Commenter l'intention, pas l'évidence
- Garder les commentaires à jour
